Building a Three-Tier Web App on AWS

A step-by-step guide to architecting a fully serverless web application using DynamoDB, Lambda, API Gateway, S3, and CloudFront.

⏱ ~150 min 💰 $0 cost 🔥 Intermediate

Three-tier architecture is a tried-and-true way of organizing web applications. Instead of cramming all your code into one place, it splits things up into three distinct layers: the presentation tier (what the user sees), the logic tier (the brains that process requests), and the data tier (the database). Each layer has a clear responsibility, making the whole system easier to debug, scale, and maintain.

Think of it like a restaurant: the front-of-house staff handle customer interaction, the kitchen handles order preparation, and the pantry stores all the ingredients. In our case, those roles are played by S3/CloudFront, Lambda/API Gateway, and DynamoDB respectively.

Tier 1

Presentation Tier — S3 + CloudFront

The presentation tier handles everything the user directly sees and interacts with. Our goal here is to get a static website hosted on S3 and delivered globally through CloudFront.

Step 1 — Create an S3 Bucket

S3 (Simple Storage Service) acts as a scalable storage container for your website files. Think of it as a highly available hard drive in the cloud — it holds your HTML, CSS, and JavaScript until CloudFront needs to serve them.

Action

Navigate to the S3 console and create a new bucket. Bucket names must be globally unique across all of AWS, since they become part of the URL when static website hosting is enabled. Use a name like yourname-threetier-abc123.

Make sure you select the AWS region closest to you.

Step 2 — Upload Website Files

Download the three website files for this project: index.html, style.css, and script.js. Each file serves a distinct purpose:

  • index.html — the structural blueprint of the page
  • style.css — fonts, colours, layout, and all visual presentation
  • script.js — handles user interactions and API calls
💡 Static vs. Dynamic Websites A static site runs its logic entirely in the browser — no backend calls needed. A dynamic site communicates with servers, databases, or APIs. Right now we're uploading files to S3, but since our app will eventually call Lambda and DynamoDB, it's a dynamic web application — we're just setting up the frontend layer first.

Step 3 — Create a CloudFront Distribution

Amazon CloudFront is a Content Delivery Network (CDN). Rather than serving your files from a single S3 bucket in one region, CloudFront caches copies of your content at edge locations around the world. The result: your users get fast load times regardless of where they're located.

Configuration

In the Origin domain field, select your S3 bucket. Under the origin access settings, allow CloudFront to access your private S3 bucket — this gives CloudFront a dedicated "key" to fetch your files without making the bucket publicly accessible.

Leave the remaining settings at their defaults and click Create distribution.

Once the distribution deploys, copy its domain name and open it in a browser. You should see your website loading over CloudFront. The presentation tier is complete!

Tier 2

Logic Tier — Lambda + API Gateway

The logic tier is where requests get processed. We'll use AWS Lambda to run code without provisioning any servers, and then place Amazon API Gateway in front of it to create a clean HTTP interface for the frontend to talk to.

Step 4 — Create a Lambda Function

AWS Lambda is a serverless compute service. You write your code, upload it, and Lambda handles everything else — provisioning machines, scaling up under load, and scaling back down when traffic drops. You're only billed for actual execution time.

Action

Head to the Lambda console and create a new function. Select Author from scratch, choose Node.js as the runtime, and give it a descriptive name like getUserData.

Step 5 — Write the Lambda Function Code

The function will accept a userId parameter, look it up in a DynamoDB table, and return the matching record. Paste the following code into the inline editor, replacing the placeholder region with your own (e.g. us-east-1):

// Import individual components from the DynamoDB client package
import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import { DynamoDBDocumentClient, GetCommand } from "@aws-sdk/lib-dynamodb";

const client = new DynamoDBClient({ region: "YOUR_REGION" });
const docClient = DynamoDBDocumentClient.from(client);

export const handler = async (event) => {
  const userId = event.queryStringParameters?.userId;
  const params = {
    TableName: "Users",
    Key: { userId }
  };
  try {
    const data = await docClient.send(new GetCommand(params));
    return {
      statusCode: 200,
      // Allow CORS for all origins, replace '*' with specific domain in production
      headers: { "Access-Control-Allow-Origin": "YOUR_CLOUDFRONT_DOMAIN" },
      body: JSON.stringify(data.Item)
    };
  } catch (err) {
    return { statusCode: 500, body: JSON.stringify({ error: err.message }) };
  }
};
💡 AWS SDK in Lambda Lambda functions interact with other AWS services through the AWS SDK. The SDK provides pre-built libraries for services like DynamoDB, S3, and SQS — so instead of writing raw HTTP calls, you import a purpose-built client and call readable methods like GetCommand.

Click Deploy to save and activate the function. Watch for the "Deployment successful" confirmation in the bottom-right corner.

Step 6 — Create an API Gateway REST API

An API (Application Programming Interface) is a contract that lets two software systems communicate. In our case, the browser needs a structured way to call Lambda — and that's exactly what API Gateway provides. It acts as the "front door", receiving HTTP requests from the website, routing them to Lambda, and passing the response back.

Configuration

In the API Gateway console, choose REST API → Build. Set the endpoint type to Regional (best for single-region apps with low latency requirements). Name the API something like UserDataAPI.

Step 7 — Add a Resource and GET Method

Resources represent the URL paths within your API (e.g. /users). Methods define what actions are allowed on that path — in our case, a GET request to retrieve user data.

Action

Create a new resource named users with path /users. Then create a GET method on that resource, setting the integration type to Lambda Function and enabling Lambda Proxy Integration. Select your getUserData function.

💡 What is Lambda Proxy Integration? Normally, API Gateway has to reformat the incoming request before Lambda can understand it, and then reformat the response before sending it back. With proxy integration enabled, API Gateway simply passes the entire raw request straight to Lambda and expects Lambda to construct the full HTTP response itself — including status codes and headers.

Step 8 — Deploy the API

In API Gateway, changes aren't live until you explicitly deploy them to a stage. Think of stages as environments: you might have dev, staging, and prod. For now, we'll deploy directly to a prod stage.

Action

Go to Actions → Deploy API. Create a new stage called prod and confirm. Once deployed, you'll see an Invoke URL — this is the public endpoint for your API.

⚠ Expected error Opening the invoke URL right now will return an error — that's perfectly normal. The DynamoDB table doesn't exist yet. We'll fix that in the next tier.
Tier 3

Data Tier — DynamoDB

The data tier is responsible for storing and retrieving the application's data. We'll use Amazon DynamoDB — a fully managed NoSQL database that's built for low-latency access at any scale.

Step 9 — Create a DynamoDB Table

DynamoDB organizes data differently from a traditional relational database. Rather than rows in a fixed schema, it stores flexible items (think JSON objects). Each item needs a unique partition key, which DynamoDB uses to distribute data across its internal servers for fast lookups.

Configuration

Name the table Users. Set the partition key to userId with type String. Leave all other settings at their defaults and click Create table.

Important: Create this table in the same region as your Lambda function.

Step 10 — Add Sample Data

While the table is being created, let's talk about what makes DynamoDB distinctive. It's schema-less — you can add items with completely different attribute sets without modifying a table structure first. This flexibility is one of the key advantages of NoSQL databases.

Action

Once the table status shows Active, open it and go to Explore table items → Create item. Switch to JSON view and turn off "DynamoDB JSON" so we can work with standard JSON. Paste the following:

{
  "userId": "1",
  "name": "Alex Johnson",
  "email": "alex@example.com"
}

Step 11 — Grant Lambda Permission to Read DynamoDB

By default, Lambda functions can't access other AWS services — you have to explicitly grant permissions through IAM. This is the principle of least privilege in action: every service only gets the access it genuinely needs.

Action

Open your Lambda function, go to the Configuration → Permissions tab, and click on the execution role name. This takes you to the IAM console. Attach the AmazonDynamoDBReadOnlyAccess policy (or AmazonDynamoDBFullAccess for this tutorial).

💡 A note on permissions AmazonDynamoDBFullAccess grants read and write access — handy for development, but too permissive for production. In a real application, you'd create a custom IAM policy scoped to only the specific table and operations your function requires.
Final Step

Connecting All Three Tiers

All the pieces are in place. Now we just need to wire them together so the frontend can talk to the backend, and the backend can talk to the database.

Step 12 — Update script.js with Your API Endpoint

Your website's script.js currently has a placeholder URL where the API Gateway invoke URL should go. We need to replace that placeholder with your actual endpoint.

Action

Download script.js from your S3 bucket. Open it locally and find line 9, which references the API URL. Replace the placeholder with your API Gateway invoke URL — it should look like https://xxxxxxxx.execute-api.REGION.amazonaws.com/prod/users.

Re-upload the updated file to your S3 bucket, replacing the original.

Step 13 — Fix the CORS Error

Once the updated file is live, visit your CloudFront URL and try looking up a user. You'll likely hit a CORS error. Here's why:

CORS (Cross-Origin Resource Sharing) is a browser security feature that prevents websites from making requests to a different domain without explicit permission. Your frontend is hosted at a CloudFront URL, but it's calling API Gateway at a completely different domain. The browser blocks this by default.

⚠ CORS error in the browser Open your browser's developer tools (press F12) and check the Console tab. You'll see a message like: "Access to fetch at 'https://…' from origin 'https://…cloudfront.net' has been blocked by CORS policy."

The fix requires two changes: enabling CORS in API Gateway and adding the correct response headers inside the Lambda function (because we're using proxy integration, Lambda is responsible for constructing the full HTTP response).

Enable CORS in API Gateway

Action

In API Gateway, select your /users resource and choose Actions → Enable CORS. Under Access-Control-Allow-Origin, enter your CloudFront domain name (e.g. https://d1xxxxxxxx.cloudfront.net). Check both GET and OPTIONS methods, then confirm.

Redeploy the API to the prod stage for the changes to take effect.

Add CORS Headers to the Lambda Response

Action

Update your Lambda function to include the Access-Control-Allow-Origin header in every response. Replace the * wildcard with your specific CloudFront domain for better security:

headers: {
  "Access-Control-Allow-Origin": "https://YOUR_CLOUDFRONT_DOMAIN",
  "Content-Type": "application/json"
},

Click Deploy to save the updated Lambda code.

Step 14 — Test the Completed Application

With CORS resolved, head back to your CloudFront URL, enter 1 in the userId field, and click the search button. You should see the user data fetched directly from DynamoDB, passed through Lambda, routed via API Gateway, and displayed on your website.

✅ All three tiers are live! Your S3/CloudFront frontend is calling API Gateway, which triggers Lambda, which reads from DynamoDB — and the response flows all the way back to the browser.

Cleanup — Avoid Ongoing Charges

Once you've verified everything is working, delete the resources to prevent unexpected AWS bills. Work through them in this order:

  • CloudFront: Disable the distribution first, wait for the status to change to Disabled, then delete it. If you delete S3 first, the distribution will point to nothing and may cause errors during deletion.
  • S3: Empty the bucket (delete all objects), then delete the bucket itself.
  • API Gateway: Delete the REST API from the API Gateway console.
  • Lambda: Delete the getUserData function.
  • DynamoDB: Delete the Users table. Choose to keep CloudWatch alarms to avoid additional charges.

Project Complete 🎉

You've built and deployed a fully serverless three-tier web application on AWS — from static hosting through to live database queries. That's a genuinely impressive architecture to have under your belt.
